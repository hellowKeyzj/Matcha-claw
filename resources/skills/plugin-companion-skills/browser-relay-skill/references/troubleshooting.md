# Browser Relay 排障参考

## 先分清是哪一段出问题

| 现象 | 先查哪一段 |
|------|-----------|
| `status` 不通 | 插件 / relay / 扩展连接 |
| 浏览器在线但没法操作页面 | 默认目标 / `targetId` / `Use This Window` |
| 页面结构拿不到 | 页面没加载完 / 当前 target 不对 / 站点限制 |
| `act` 失败 | ref 失效 / 页面已变 / 需要先等待 |
| 页面和用户看到的不一致 | 目标窗口选择错误，或 `open` / `focus` 后页面已切换 |
| relay 不通但 direct-cdp 可用 | 扩展链路问题，不是浏览器引擎问题 |

## 安装与连接问题

### 现象：扩展没有连上

优先确认：

1. Browser Relay 模式已启用
2. Chrome 已安装 MatchaClaw Browser Relay 扩展
3. 用户点击过扩展图标
4. badge 不是 `!`
5. 目标窗口点过 **Use This Window**

### 现象：没有当前可执行页面

这通常不是“browser 工具坏了”，而是默认目标为空。

优先处理：

1. 让用户在目标窗口点一次 **Use This Window**
2. 确认该窗口里确实有正常网页，而不是 `chrome://`、扩展页之类内部页
3. 再跑 `tabs`

## 目标页错误

### 现象：操作打到了不该打的页面

优先判断：

- 是否错误复用了默认窗口
- 是否应该显式传 `targetId`
- 是否应该先 `open` 自己的工作页，而不是继续用用户当前页

### 现象：`snapshot` 后 ref 失效

常见原因：

- 页面发生重渲染
- 列表刷新
- 切页或展开后结构变了

处理方式：

1. 重新 `snapshot`
2. 必要时先 `highlight`
3. 再执行 `act`

## relay 与 direct-cdp

### 什么时候坚持 relay

- 正常网页任务
- 需要登录态
- 需要真实用户浏览器环境
- 用户没有明确要求切模式

### 什么时候短暂切 direct-cdp

- 在排障浏览器连接层
- 用户明确要求
- relay 扩展链路暂时不可用，需要确认浏览器是否仍可直连

### 注意

`direct-cdp` 是排障路径，不要把它写成默认工作流。

## 页面动作失败

### `click` / `type` / `fill` 失败

按这个顺序查：

1. ref 是否还有效
2. 元素是否在当前视口 / 当前 frame
3. 页面是否还没加载完
4. 是否应该先 `wait`
5. 是否应该换成更接近真实交互的动作路径

### `upload` 失败

优先确认：

1. 目标是不是 file input
2. `inputRef` / `element` 是否命中正确
3. 路径是否真实存在

### `screenshot` / `pdf` 没保存到预期位置

优先使用工作区内相对 `savePath`。  
如果传了不合适的外部路径，保存结果可能不会按你预期落盘。

## 诊断动作建议

### 页面行为不对

优先动作：

- `snapshot`
- `errors`
- `requests`
- 必要时 `console` / `act -> evaluate`

### 判断站点是不是拦了你

看这些信号：

- 页面结构明显不完整
- 请求报 401 / 403 / challenge
- 页面文本出现登录墙、风控提示
- 同样 URL 在用户看到的页面和工具拿到的页面结果不一致

## 收口原则

- 修复后必须按同一条路径复验
- 不要把“页面暂时没反应”直接当成根因
- 不要因为一次失败就立刻切 `direct-cdp`
- 不要在不确认 target 的情况下连续执行多步动作
