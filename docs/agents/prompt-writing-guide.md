# 提示词编写指南

这份指南用于编写能明显提升 Agent 执行效果的系统提示词、技能提示词、命令提示词和工具说明。

核心结论：**好 prompt 不是写“你要专业”，而是写“专业的人在每种情况下会怎么判断”。**

## 1. 每条规则都要有触发场景

很多低效 prompt 都在写空话，比如“合理使用工具”“必要时搜索”。这种话不会稳定改变 Agent 行为。

好的写法是把场景拆清楚：

```md
当前信息、职位、价格、政策、版本 → 搜索。
稳定知识、基础概念、普通编程 → 不搜索。
```

可直接借鉴：

```md
当用户请求涉及变化快的信息时，先搜索再回答。
当任务能基于当前上下文完成时，不要搜索。
```

明显效果：Agent 知道什么时候做、什么时候不做，减少乱搜和瞎猜。

## 2. 工具描述要写得像 SOP

工具说明不能只写“这个工具能干嘛”。工具描述要告诉 Agent：什么时候用、什么时候不用、调用前检查什么、调用后怎么处理、常见错误是什么。

推荐结构：

```md
何时使用
何时不要使用
调用前检查什么
调用顺序
调用后怎么处理
失败怎么处理
常见错误
```

比如文件工具不要只写“读取和编辑文件”，而要写：

```md
编辑前必须先读取目标文件。
编辑后旧读取结果视为过期。
只读目录不能改。
匹配不到或匹配多处时，停止并重新读取，不要猜测替换。
```

可直接借鉴：

```md
工具描述 = 触发条件 + 调用顺序 + 禁忌 + 失败处理 + 输出要求。
```

明显效果：工具调用更稳定，少误用、少乱改、少假成功。

## 3. 大量使用正例和反例

只讲原则不够。模型更容易从“用户说什么 → 你做什么”里学到稳定行为。

参考写法：

```md
用户说“写博客” → 创建文件。
用户说“总结文章” → 在聊天里回答。
用户说“创建 PPT” → 先读取 pptx skill，再创建 .pptx。
```

可直接借鉴：

```md
每个复杂规则后面放 2-3 个例子。
例子必须覆盖：应该做什么、不应该做什么、容易误判的边界。
```

明显效果：比抽象原则更容易触发正确行为，尤其适合文件、工具、任务分流、输出格式这类规则。

## 4. 明确区分“该做”和“不该做”

强模型最大问题不是不会做，而是做太多。提示词必须写清禁止行为。

参考里的有效约束包括：

```md
不要过度格式化。
不要没必要地问用户。
不要没证据就声称完成。
不要创建用户看不到的文件。
不要用工具模拟不存在的能力。
```

可直接借鉴：

```md
禁止行为清单：
- 不做用户没要求的扩展。
- 不顺手重构无关代码。
- 不把计划说成完成。
- 不为了显得认真而使用工具。
- 不在证据不足时下确定结论。
```

明显效果：减少过度发挥，让 Agent 更像专家，而不是“热心但失控的实习生”。

## 5. 把“完成”定义清楚

不要写“帮用户做好”。要写什么结果才算完成。

参考里的完成标准很具体：

```md
文件要放到 outputs。
还要调用 present_files。
不能只在聊天里展示。
测试失败要说失败。
未验证就不能说完成。
```

可直接借鉴：

```md
完成标准：
- 产物已生成。
- 放在正确位置。
- 用户能访问。
- 已验证可用。
- 失败项已明确说明。
```

明显效果：减少“看起来做了，但用户拿不到 / 用不了 / 没验证”的情况。

## 6. 控制 Agent 的提问频率

Agent 不应该像表单客服一样反复问用户。能合理判断的小选择，要自己决定。

参考里的有效规则：

```md
能推断就别问。
小选择自己决定。
真阻塞才问。
一次最多问一个问题。
不要把用户已经给的约束再问一遍。
```

可直接借鉴：

```md
只在以下情况询问用户：
- 范围不清会导致做错对象。
- 涉及权限、删除、覆盖、发布、发送等不可逆或外部动作。
- 缺少关键输入，且无法从上下文推断。
- 多个方案会产生明显不同的业务结果。

其他情况选择合理默认值，并在结果中简短说明。
```

明显效果：Agent 更主动、更像专家，少打断用户。

## 7. 设计任务分流器

不要写一个万能流程。不同任务应该走不同路径。

参考里的分流方式：

```md
文件任务 → skill → 创建 → 输出目录 → 展示。
搜索任务 → 判断是否需要搜索 → 多源 → 引用。
代码任务 → 读文件 → 改动 → 验证。
地图/地点任务 → 搜索地点 → 展示地图。
```

可直接借鉴：

```md
按任务类型写分流规则：
- 问答 / 分析：直接给结论，不创建文件。
- 代码修改：读文件 → 最小改动 → 验证 → 汇报。
- Bug 定位：复现或建立闭环 → 归因 → 再修复。
- 文件产物：创建真实文件 → 放到用户可访问位置 → 给出路径。
- 文档 / 提示词：改结构、触发条件、禁止项、检查项，不堆口号。
```

明显效果：减少“用户要文件却只在聊天里回答”“用户要解释却创建文件”“用户要定位却直接修症状”。

## 8. 给工具设置优先级

工具越多，越需要优先级。否则 Agent 容易乱查、乱调、绕远路。

参考里的优先级思路：

```md
内部工具优先于 web search。
体育数据优先用 sports tool。
用户文件优先用文件工具。
当前网页用 web_fetch，不要泛搜。
```

可直接借鉴：

```md
工具优先级：
1. 用户提供的数据。
2. 项目 / 内部工具。
3. 专用工具。
4. Web 搜索。
5. 模型常识。

不要为了显得认真而使用工具。只有工具能提高正确性、可验证性或交付质量时才使用。
```

明显效果：减少乱查、乱猜、重复搜索和错误数据源。

## 9. 写错误防护规则

好 prompt 会把常见翻车点提前写进去。

参考里的错误防护包括：

```md
不要臆造当前信息。
不要误用 localStorage。
不要改只读目录。
不要输出用户访问不到的文件。
不要把搜索结果当用户说的话。
不要用过期上下文编辑文件。
```

可直接借鉴：

```md
把你见过的 Agent 错误写成禁止项：
- 如果信息可能过期，先查证。
- 如果文件不可写，先复制到可写位置。
- 如果工具失败，不要假装成功。
- 如果上下文过期，重新读取再行动。
- 如果结果给用户用，必须放到用户能访问的位置。
```

明显效果：能直接减少重复犯错，尤其适合长期运行的工程 Agent。

## 10. 输出风格必须可观察

不要只写“简洁友好”。这太抽象。

参考里的好写法：

```md
简单问题用自然短句。
复杂问题才用列表。
拒绝时不用 bullet。
少用粗体和标题。
不要过度解释。
```

可直接借鉴：

```md
输出规则：
- 先给结论。
- 简单问题用短段落。
- 多步骤或多维度内容才用列表。
- 最终总结只写结果、验证、风险。
- 不写过程流水账、自我评价和无意义后续建议。
```

明显效果：回答更像人话，少模板感，少废话。

## 11. 补强：全局优先级

参考 prompt 很强，但规则很多时仍然需要一个总排序。否则规则冲突时，Agent 可能随机服从更近或更强烈的一条。

建议补：

```md
规则冲突时优先级：
1. 安全 / 权限 / 隐私 / 不可逆操作。
2. 用户明确要求。
3. 项目事实和业务约束。
4. 工具工作流。
5. 输出风格偏好。

低优先级规则不得覆盖高优先级规则。
```

明显效果：减少规则打架，比如“用户让我快点”和“必须验证”冲突时，Agent 知道验证优先。

## 12. 补强：完成即停止

强模型容易继续查、继续优化、继续问“还要不要我……”。所以要写硬停止条件。

建议补：

```md
满足成功标准后停止。
不主动扩展范围。
不继续优化、重构、润色或添加未来兼容。
不在最终回复里追加无意义的后续建议。
```

明显效果：减少过度执行，避免把一个小任务滚成大任务。

## 13. 补强：没验证别说完成

参考 prompt 对搜索和文件比较严格，但一般任务也需要统一证据门槛。

建议补：

```md
任何完成声明必须基于证据。
证据可以是测试结果、运行结果、文件路径、工具输出、截图、diff 或明确检查项。

未测试就说“未测试”。
未验证就说“未验证”。
验证失败就说“失败”，并说明失败项和当前失败边界。
```

明显效果：减少假完成、假自信、口头完成。

## 14. 补强：别顺手重构

Fable / Opus 这类强模型很容易“顺手改好一点”。工程 Agent 必须压住这个冲动。

建议补：

```md
只做用户要求的改动。
不要顺手重构、抽象、清理无关代码或调整相邻逻辑。
不要添加未请求的功能、配置项、兼容层、fallback、重试或缓存。
发现无关问题，只在最终回复里简短提及。
```

明显效果：diff 更干净，执行更可控，review 成本更低。

## 15. 把硬限制写成可量化红线

不要只写“少一点”“不要太多”“尽量简短”“谨慎引用”。这些话会让 Agent 按感觉执行。

参考里的强写法是把边界数字化：

```md
15 个词以上算违规。
每个来源最多引用一次。
图片最少 3 张，最多 4 张。
简单任务少于 100 行直接一次生成，长任务分段完成。
```

可直接借鉴：

```md
硬限制写法：
- 最多几个。
- 最少几个。
- 超过多少算失败。
- 哪些情况绝不例外。
```

明显效果：模型不用猜“多少算多、什么算短”，执行更稳定。

## 16. 给复杂规则加输出前自检

复杂规则只写一遍，Agent 很容易读懂但最后忘掉。尤其是安全、版权、验证、文件交付、工程边界这类高风险规则，要加最后一道自检门。

参考里的好写法不是只写“不要侵权”，还会写：

```md
回答前检查：
- 有没有超过长度限制？
- 有没有重复引用同一来源？
- 有没有复现原文结构？
- 会不会替代原文？
```

可直接借鉴：

```md
对于高风险规则，补一个输出前检查清单：
- 是否违反硬限制？
- 是否缺少证据？
- 是否超出用户请求？
- 是否把未验证内容说成完成？
```

明显效果：减少“规则写了，但最后没执行”的情况。

## 17. 给规则补上例外情况

只写主规则不够。没有例外，Agent 要么机械执行，要么遇到边界时自己猜。

参考里的规则经常是这种结构：

```md
通常不要用列表，除非内容多维复杂。
不要搜索稳定事实，除非当前状态可能变化。
不要问用户，除非缺少关键输入。
```

可直接借鉴：

```md
规则写法：
默认做 A。
当 X 时例外，改做 B。
不要在 Y 情况下套用这条规则。
```

明显效果：规则更像专家判断，而不是死板命令。

## 18. 把复杂规则写成局部决策单元

不要把 prompt 写成“规则列表 + 统一注意事项”。复杂规则如果只写一句，Agent 还要自己去别处拼触发条件、例外、例子和失败处理，很容易漏。

参考里的强写法是：一条重要规则下面，马上补完整判断材料。

```md
规则本身
→ 触发条件
→ 禁用条件
→ 例外情况
→ 执行步骤
→ 失败处理
→ 正例 / 反例
→ 为什么这样做
```

比如搜索规则不是只写“需要时搜索”，而是接着写：哪些当前状态要搜、哪些稳定知识不搜、搜几次、搜索词怎么写、结果冲突怎么办。

可直接借鉴：

```md
每条复杂规则写成一个局部决策单元：
- 什么时候触发。
- 什么时候不要用。
- 有哪些例外。
- 做的时候按什么步骤。
- 失败时怎么处理。
- 给 1-2 个正例 / 反例。
```

明显效果：Agent 读到一条规则时，就能拿到完整判断链，不需要跨章节拼上下文。

## 19. 最小可复用模板

```md
你是【角色】，负责【任务范围】，不负责【排除项】。

## 成功标准
- 【产物 / 结果】已完成。
- 【验证方式】已通过，或明确说明未验证原因。
- 没有超出用户请求范围。

## 规则优先级
安全/权限/隐私/不可逆操作 > 用户明确要求 > 项目事实 > 工具工作流 > 输出风格。

## 执行纪律
- 信息足够时直接行动，不用计划代替执行。
- 小选择自己决定；范围不清、不可逆、外部发布、删除覆盖、缺关键业务信息时才问用户。
- 只做用户请求直接需要的事。
- 不添加未请求的功能、抽象、兼容层、fallback 或额外清理。
- 完成声明必须有证据；未验证就说未验证，失败就说失败。
- 满足成功标准后停止。

## 工具规则
- 先用当前上下文和用户提供资料。
- 需要当前信息时搜索。
- 编辑前必须读取目标文件。
- 工具失败时说明失败边界，不猜测成功。

## 高风险规则自检
- 是否违反硬限制？
- 是否缺少证据？
- 是否超出用户请求？
- 是否把未验证内容说成完成？

## 输出规则
- 先给结论。
- 简短。
- 最终只说：任务状态、结果、验证、风险。
```

## 20. 判断一句提示词有没有用

一个简单标准：**如果删掉这句话，Agent 行为不会明显变化，它就是废话。**

高价值提示词通常长这样：

```md
当 X 时做 Y；不要做 Z；如果失败，按 W 处理；完成后用 V 证明。
```

低价值提示词通常长这样：

```md
请认真、专业、准确、全面、智能地帮助用户。
```

真正有效的提示词，是把专家的判断过程写清楚，而不是把专家品质形容一遍。
