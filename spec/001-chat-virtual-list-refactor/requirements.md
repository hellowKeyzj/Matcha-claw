# 需求文档 - 聊天页虚拟列表重构

状态：DONE

## 简介

当前聊天页已经接入虚拟列表来降低长会话渲染开销，但滚动语义还是沿用了普通 DOM 列表时期的做法。结果就是：

- 切换会话时，有时落在历史开头，有时落在中间，不稳定
- 当前会话原本在底部附近时，新消息和流式输出不一定继续吸底
- 为了兜底，页面里同时混用了 `scrollToIndex(...)`、`scrollIntoView(...)` 和多段 `requestAnimationFrame` 时序修补
- 这套逻辑现在已经出现“修一个回归，又带出另一个回归”的情况

这次 Spec 的目标不是再补一个滚动 bug，而是把聊天页按“虚拟聊天列表”重新设计，让滚动语义和渲染模型一致。

这个能力主要服务两类人：

- 日常在聊天页里切换 agent 会话、追看最新消息的用户
- 后续要继续维护聊天页滚动、流式消息和会话切换逻辑的开发者

成功后，聊天页应该具备两个直接收益：

- 用户看到的滚动行为稳定、可预期，不再靠时序兜底
- 后续开发者能在明确模型上改动，而不是继续堆 ref 和 effect

## 术语表

- **System**：MatchaClaw 渲染层聊天页
- **虚拟聊天列表**：使用 `@tanstack/react-virtual` 渲染的消息列表，只渲染可见区附近的消息行
- **聊天行**：聊天区域里真正参与垂直排布的一条记录，可能是历史消息，也可能是流式消息、处理中提示等临时行
- **吸底**：视口自动跟随到当前会话最新一条聊天行的底部
- **Sticky Bottom**：用户当前处于“希望跟随最新消息”的状态。只要这个状态成立，新消息追加后就应该继续吸底

## 范围说明

### In Scope

- 重构聊天页滚动模型，让虚拟列表成为唯一的滚动定位依据
- 将历史消息和底部临时内容整理成统一的聊天行模型
- 明确“切会话默认到最新消息底部”和“用户上翻后不强拉回底部”的语义
- 补齐覆盖虚拟列表滚动语义的单元测试和页面级回归测试

### Out of Scope

- 不改 OpenClaw Gateway 协议、事件格式和消息持久化格式
- 不改聊天消息内容本身的渲染样式、Markdown 解析和审批业务规则
- 不顺带重做任务收件箱、输入框或工具栏布局

## 需求

### 需求 1：切换会话时稳定打开到最新消息

**用户故事：** 作为正在多个 agent 会话间切换的用户，我希望点开会话后直接看到最新消息，以便继续当前对话，而不是先看到很早之前的内容。

#### 验收标准

1. WHEN 用户从左侧会话列表打开任意会话 THEN System SHALL 在该会话首屏稳定落在最新聊天行底部
2. WHEN 当前会话已有大量历史消息且启用了虚拟列表 THEN System SHALL 仍然用同样的规则打开到最新聊天行底部
3. WHEN 会话切换完成后虚拟列表还在测量高度 THEN System SHALL 在测量稳定后仍保持底部对齐，而不是回弹到开头或中间

### 需求 2：当前会话里的 Sticky Bottom 语义必须明确

**用户故事：** 作为正在连续收发消息的用户，我希望当我停在底部附近时页面继续跟随最新消息，但当我手动往上翻历史时不要被强行拉回去。

#### 验收标准

1. WHEN 用户当前位于底部附近且新增历史消息、流式消息或处理中提示追加到列表末尾 THEN System SHALL 自动继续吸底
2. WHEN 用户手动向上滚动离开底部附近 THEN System SHALL 停止自动吸底，直到用户再次回到底部附近或重新打开会话
3. WHEN 消息高度因为 Markdown、图片、工具结果或流式更新发生变化 THEN System SHALL 保持 Sticky Bottom 语义不被布局抖动错误改写

### 需求 3：聊天滚动逻辑必须建立在统一聊天行模型上

**用户故事：** 作为维护聊天页的开发者，我希望历史消息和临时底部内容使用同一套行模型参与虚拟列表排布，以便滚动定位只认一套坐标，不再混用两套机制。

#### 验收标准

1. WHEN 聊天页渲染历史消息、流式消息、处理中提示或打字提示 THEN System SHALL 先把它们整理成统一的聊天行列表，再交给虚拟列表渲染
2. WHEN 需要滚到“最后一条” THEN System SHALL 根据统一聊天行列表的最后一项做定位，而不是同时依赖 `scrollIntoView(...)` 和 `scrollToIndex(...)`
3. WHEN 后续新增一种底部临时行 THEN System SHALL 只需要扩展聊天行模型，而不需要再追加新的滚动补丁

### 需求 4：回归测试必须能锁定虚拟列表主路径

**用户故事：** 作为后续维护聊天页的开发者，我希望回归测试直接覆盖虚拟列表滚动主路径，以便再次调整聊天页时能尽早发现吸底和切会话回归。

#### 验收标准

1. WHEN 聊天滚动语义发生变更 THEN System SHALL 通过针对虚拟列表的单元测试明确暴露失败，而不是只靠手工截图发现
2. WHEN 切换会话、追加新消息、用户上翻历史这三类关键路径发生回归 THEN System SHALL 有对应测试失败
3. WHEN 测试运行于 jsdom 或 mock 虚拟列表环境 THEN System SHALL 仍能稳定表达真实滚动语义，而不是依赖偶然的 DOM 时序

## 非功能需求

### 非功能需求 1：性能

1. WHEN 会话历史消息达到数百条 THEN System SHALL 继续使用虚拟列表，只渲染可见区附近聊天行
2. WHEN 流式输出持续更新 THEN System SHALL 避免因为全量重排导致明显卡顿或滚动抖动
3. WHEN 本次重构完成 THEN System SHALL 不回退到全量消息渲染，并且切会话与流式更新链路不新增整页级重复渲染热点

### 非功能需求 2：可靠性

1. WHEN 历史消息、流式消息和临时底部行在短时间内连续更新 THEN System SHALL 保持滚动定位规则一致，不依赖多段时序补丁猜测结果
2. WHEN 页面切换会话、恢复运行中会话或重新拉历史 THEN System SHALL 不出现“首屏在顶部，但下一次又到底部”的随机行为

### 非功能需求 3：可维护性

1. WHEN 后续开发者继续调整聊天滚动逻辑 THEN System SHALL 能在单一滚动模型和单一聊天行模型上扩展，而不是修改多个分散的 ref 和 effect
2. WHEN 需要排查滚动回归 THEN System SHALL 能通过明确状态和测试快速定位，而不是靠逐帧猜测浏览器时序

## 成功定义

- 聊天页切会话和新消息追加的滚动行为，能被稳定描述为一套统一规则
- 删除当前依赖混合定位和多段时序兜底的临时修补逻辑
- 聊天页相关回归测试可以直接覆盖“切会话到底部”“底部附近继续吸底”“用户上翻后停止吸底”三条主路径
- 保留虚拟列表性能收益，不把聊天页回退成未优化前的全量消息渲染模型
