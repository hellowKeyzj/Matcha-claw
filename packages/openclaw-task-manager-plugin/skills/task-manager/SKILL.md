# Task Manager 协议（框架版）

## 1. 触发条件（先做复杂度评估）

当请求满足任一条件，优先进入任务模式：

1. 需要 3 步及以上才能完成。
2. 存在明确顺序依赖（先/再/然后/最后）。
3. 任务可能跨轮次、跨会话或需要中断恢复。
4. 任务执行中可能出现用户确认或外部审批节点。

复杂度评估维度（必须显式思考）：

- 步骤规模：是否多步骤、可否一次完成。
- 依赖顺序：是否必须按阶段推进。
- 中断恢复：是否需要暂停后继续。

## 2. 主会话协议（调度）

### 阶段 A：规划与确认

1. 先输出 Markdown 计划草案，不直接执行业务工具。
2. 询问用户是否按此计划执行。
3. 用户确认后进入阶段 B。

### 阶段 B：创建与委派

按固定顺序调用：

1. `task_create`
2. `task_set_plan_markdown`
3. `sessions_spawn`（必须包含 `mode: "run"`、`cleanup: "delete"`）
4. `task_bind_session`

## 3. 子会话协议（执行）

1. 仅围绕单个 `taskId` 执行。
2. 每完成一个步骤，调用 `task_set_plan_markdown`（传完整 Markdown）。
3. 出现“当前回合内无法自行决策”的节点时，优先判定阻塞类型：
   - 需用户当下决策：走 `task_request_user_input`。
   - 需等待异步外部信号：走 `task_wait_approval`。
4. 挂起后必须结束当前回合，禁止继续调用普通业务工具。
5. 无法继续时，调用 `task_mark_failed` 并说明原因。
6. 任务完成后输出摘要并结束当前回合。

### 3.1 `task_request_user_input` 调用协议（同步人工决策）

触发条件（命中任一）：

1. 操作具有不可逆影响，且需要用户明确授权（如覆盖/删除/提交）。
2. 存在互斥路径，需要用户在当前回合内二选一或多选一。
3. 缺少关键业务参数，且参数只能由当前用户补充。

调用前要求：

1. 先用 `task_set_plan_markdown` 落盘当前进度，确保当前步骤保持 `- [ ]`。
2. 问题必须是单一决策点，避免一个问题混入多个判断。

调用后行为：

1. 调用 `task_request_user_input(taskId, question)`。
2. 输出“等待用户确认”的简短说明后，立即结束当前回合。
3. 不得在同一回合继续执行后续步骤。

恢复方式：

1. 由 UI 或主会话调用 `task_resume` 写回用户决策。
2. 恢复后从原阻塞步骤继续推进，不得跳步。

### 3.2 `task_wait_approval` 调用协议（异步外部信号）

触发条件（核心定义）：

若任务推进依赖异步外部信号（人/系统/回调/定时事件），且当前回合无法立即闭环，调用 `task_wait_approval` 挂起任务并结束当前回合；收到信号后通过 `task_resume` 恢复执行。

典型场景：

1. 外部系统回调（支付、签署、审批网关、风控结果）。
2. 跨系统状态轮询（工单、合同、物流、批处理完成）。
3. 时间触发事件（T+1、定时任务完成、窗口期到达）。

调用前要求：

1. 先用 `task_set_plan_markdown` 记录“等待信号”状态，并保持当前步骤 `- [ ]`。
2. `description` 要明确等待对象与恢复条件（谁、什么信号、何时继续）。

调用后行为：

1. 调用 `task_wait_approval(taskId, description)`。
2. 输出“已挂起，等待异步信号”的说明后，结束当前回合。
3. 禁止伪造完成状态，禁止把未完成步骤标记为 `- [x]`。

恢复方式：

1. 推荐由 webhook/系统事件触发 `task_resume`（自动恢复）。
2. 也可由人工在主会话显式携带 `taskId` 触发恢复（手动恢复）。

### 3.3 误用防护

1. 需要“当下用户决策”却使用 `task_wait_approval`，会导致交互链路变长。
2. 需要“异步外部信号”却使用 `task_request_user_input`，会造成任务假阻塞在会话层。
3. 仅在文本写“受阻”，但把步骤全勾选为 `- [x]`，属于状态违规。

## 4. 动态纠错协议（执行中途可切换）

若执行过程中发现“步骤膨胀”或“依赖复杂化”，允许中途切换到任务模式：

1. 暂停继续堆叠普通工具调用。
2. 立即整理当前已完成/待完成步骤为 Markdown 计划。
3. 调用 `task_create` + `task_set_plan_markdown`，并委派子会话接管。

原则：允许反悔，禁止硬撑。

## 5. Markdown 规范（用于可视化）

1. 顶层步骤必须是顶格 checkbox：
   - `- [ ] 步骤1: 标题`
   - `- [x] 步骤2: 标题`
2. 子步骤必须缩进两个空格并使用 checkbox：
   - `  - [ ] 子步骤A`
3. 说明与证据使用前缀，避免与子步骤混淆：
   - `  - 说明: ...`
   - `  - 完成情况: ...`
   - `    - 证据: ...`
4. `task_set_plan_markdown` 必须提交完整文档，不提交增量片段。
